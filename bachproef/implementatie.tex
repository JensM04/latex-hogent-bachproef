\chapter{\IfLanguageName{dutch}{Implementatie}{Implementation}}%

Dit hoofdstuk beschrijft de concrete uitwerking van het proof of concept dat werd ontwikkeld in het kader van dit onderzoek. Zoals beschreven in de methodologie, werd de implementatie stapsgewijs opgebouwd: eerst een werkende basisstructuur, vervolgens de integratie van de AR-laag. Dit hoofdstuk volgt die logische opbouw en documenteert per component de technische keuzes, de implementatiedetails en de aansluiting op de in de literatuurstudie beschreven vereisten.

\section{Technische omgeving en tools}

De applicatie werd ontwikkeld als een native iOS-applicatie in de programmeertaal Swift (versie 5.9), binnen de ontwikkelomgeving Xcode 15. De keuze voor een native iOS-oplossing is gemotiveerd door drie factoren: de rijke ondersteuning van Apple voor AR-toepassingen via het ARKit-framework, de stabiele integratie met de hardware van de iPhone (camera, GPS, gyroscoop, versnellingsmeter), en de betere prestatiegaranties in vergelijking met cross-platform alternatieven zoals React Native of Flutter. ARKit vereist directe toegang tot de AVCaptureSession van het toestel, wat in een cross-platform omgeving doorgaans extra abstractielagen introduceert die de latentie verhogen.

De voornaamste frameworks die gebruikt worden, zijn:

\begin{itemize}
    \item \textbf{ARKit} – Apple's framework voor augmented reality op iOS. Verantwoordelijk voor de world tracking-sessie, de koppeling tussen de camerafeed en het 3D-coördinatenstelsel, en de verwerking van sensordata (camera, versnellingsmeter, gyroscoop).
    \item \textbf{SceneKit} – het 3D-renderingframework van Apple, gebruikt voor het bouwen en animeren van de AR-knooppunten: de navigatiepijl, de grafmarker en de billboard-labels.
    \item \textbf{MapKit} – het kaartframework van Apple, gebruikt voor de minikaart-component. Biedt satellietweergave, annotatiebeheer en routeweergave.
    \item \textbf{CoreLocation} – verantwoordelijk voor de GPS-positionering van de gebruiker en de kompasrichting. Levert coördinaten in het WGS-84-stelsel en de ware-noorden-heading in graden.
    \item \textbf{Combine} – Apple's reactief programmeerframework, gebruikt voor het publiceren van statuswijzigingen (geselecteerd graf, afstand tot doel, heading) naar de UI zonder expliciete callback-ketens.
\end{itemize}

De applicatie maakt gebruik van het SwiftUI app lifecycle-model als startpunt, maar de volledige interface is geïmplementeerd in UIKit. De brug tussen beide werelden wordt gelegd via een \texttt{UIViewControllerRepresentable}-wrapper:

\begin{listing}[H]

            var body: some Scene {
                WindowGroup {
                    RootViewControllerRepresentable()
                    .ignoresSafeArea()
                    .statusBarHidden(true)
                }
            }
        }
        
        struct RootViewControllerRepresentable: UIViewControllerRepresentable {

            }

    \end{minted}
    \caption[SwiftUI naar UIKit bridge]{Bridge tussen SwiftUI lifecycle en UIKit root controller.}
\end{listing}

Deze aanpak laat toe om ARKit's camerasessie onmiddellijk te starten bij het openen van de applicatie, zonder enige SwiftUI-navigatiewrapper die de eerste frame vertraagt. Dit sluit aan bij de bevinding dat het verminderen van cognitieve belasting een kernvereiste is voor bruikbare AR-navigatiesystemen \autocite{Zhao2025}.

\section{Softwarearchitectuur}

De architectuur van de applicatie is opgesplitst in afzonderlijke lagen, overeenkomstig het Repository-patroon en het principe van \textit{separation of concerns}. Elke laag is verantwoordelijk voor één domein en communiceert met andere lagen via Swift-protocollen (delegates). Dit maakt de code testbaar en uitbreidbaar: de databron kan bijvoorbeeld vervangen worden door een live REST API zonder wijzigingen in de UI-code. Dit kan belangrijk zijn voor toekomstig gebruik van de app door bijvoorbeeld APIs te gebruiken die data ophalen van verschillende gemeenten.

\subsection{Datamodellen (Models.swift)}

Het centrale model is \texttt{GraveRecord}, een Swift-struct die alle informatie over één graf bevat:

\begin{listing}[H]
    \begin{minted}{swift}
        struct GraveRecord: Identifiable, Codable, Hashable {
            let id: String
            let firstName: String
            let lastName: String
            let birthYear: Int?
            let deathYear: Int?
            let biography: String?
            let latitude: Double
            let longitude: Double

                    case let (nil, d?): return "d. \(d)"

                }
            }

Ten slotte bevat het bestand een uitbreiding op \texttt{CLLocationCoordinate2D} die de \texttt{bearing()}-methode implementeert. Deze berekent de kompasrichting in graden (0–360, met de klok mee vanaf ware noorden) van één GPS-coördinaat naar een ander, op basis van de sferische cosinusregel:

\begin{listing}[H]
    \begin{minted}{swift}
        extension CLLocationCoordinate2D {
            func bearing(to other: CLLocationCoordinate2D) -> Double {
                let lat1 = self.latitude  * .pi / 180
                let lat2 = other.latitude * .pi / 180
                let dLon = (other.longitude - self.longitude) * .pi / 180
                
                let y = sin(dLon) * cos(lat2)
                let x = cos(lat1) * sin(lat2) - sin(lat1) * cos(lat2) * cos(dLon)
                
                let bearing = atan2(y, x) * 180 / .pi
                return (bearing + 360).truncatingRemainder(dividingBy: 360)
            }
        }
    \end{minted}
    \caption[Bearing-berekening]{Bearing-berekening tussen twee GPS-coördinaten.}
\end{listing}

\subsection{Dataservice (GraveDataService.swift)}

De \texttt{GraveDataService}-klasse volgt het Repository-patroon en fungeert als enige bron van waarheid voor grafdata. Bij het opstarten laadt de service de bundled \texttt{graves.json} asynchroon op een achtergrondthread, om blokkering van de UI-thread te vermijden:

\begin{listing}[H]
    \begin{minted}{swift}
        func load() async {
            isLoading = true
            defer { isLoading = false }
            
            guard let url = Bundle.main.url(forResource: "graves",
            withExtension: "json") else { return }
            
            let graves = try await Task.detached(priority: .userInitiated) {
                let data = try Data(contentsOf: url)
                return try JSONDecoder().decode([GraveRecord].self, from: data)
            }.value
            
            self.allGraves = graves
        }
    \end{minted}
    \caption[Asynchroon laden van grafdata]{Asynchroon laden van grafdata op een achtergrondthread.}
\end{listing}

De zoekfunctie past diakritiek-insensitieve vergelijking toe. Dit is bijzonder relevant voor Belgisch kerkhofdata, waar namen frequent accenten bevatten (é, ë, ü, ç):

\begin{listing}[H]
    \begin{minted}{swift}
        func search(query: String) -> [GraveRecord] {
            let options: String.CompareOptions =
            [.caseInsensitive, .diacriticInsensitive]
            
            return allGraves
            .filter { grave in
                grave.fullName.range(of: query, options: options) != nil
                || grave.section.range(of: query, options: options) != nil
            }
            .sorted { a, b in
                let aPrefix = a.fullName.range(
                of: query, options: [options, .anchored]) != nil
                let bPrefix = b.fullName.range(
                of: query, options: [options, .anchored]) != nil
                if aPrefix != bPrefix { return aPrefix }
                return a.lastName < b.lastName
            }
        }
    \end{minted}
    \caption[Diakritiek-insensitieve zoekfunctie]{Zoekfunctie met diakritiek-insensitieve matching en relevantiesortering.}
\end{listing}

Resultaten worden gesorteerd op relevantie: prefixmatches (namen die beginnen met de zoekopdracht) krijgen hogere prioriteit dan middenin-matches.

\section{De AR-component (ARSessionManager.swift)}

De \texttt{ARSessionManager}-klasse vormt de kern van de technische bijdrage van dit proof of concept. Ze beheert de ARKit-sessie, converteert GPS-coördinaten naar AR-wereldcoördinaten, bouwt de 3D-navigatiepijl en de 3D-grafmarker, en voert per-frame updates uit van de pijlrichting.

\subsection{ARKit-sessieconfiguratie}

De ARKit-sessie wordt geconfigureerd met \texttt{ARWorldTrackingConfiguration}:

\begin{listing}[H]
    \begin{minted}{swift}
        private let configuration: ARWorldTrackingConfiguration = {
            let c = ARWorldTrackingConfiguration()
            c.worldAlignment = .gravityAndHeading
            c.planeDetection = .horizontal
            return c
        }()
    \end{minted}
    \caption[ARKit sessieconfiguratie]{Configuratie van de ARKit world tracking sessie.}
\end{listing}

De cruciale instelling is \texttt{worldAlignment = .gravityAndHeading}. Dit uitlijningssysteem zorgt ervoor dat:
\begin{itemize}
    \item de Y-as samenvalt met de zwaartekracht (omhoog);
    \item de negatieve Z-as wijst naar magnetisch noorden;
    \item de X-as wijst naar het oosten.
\end{itemize}

Dit stemt overeen met het WGS-84 geografische coördinatenstelsel en maakt directe conversie van GPS-coördinaten naar AR-wereldposities mogelijk zonder extra rotatieberekeningen.

\subsection{GPS naar AR-coördinaten: de vlakke-Aarde-benadering}
\label{sec:gpsconversie}

Een centrale technische uitdaging in buitenomgevings-AR is de omzetting van WGS-84 GPS-coördinaten naar het lokale Cartesische coördinatenstelsel van ARKit. De in deze implementatie gebruikte methode is de \textit{vlakke-Aarde-benadering}:

\[
X_{\text{AR}} = \Delta\text{lon} \times 111{.}320 \times \cos(\varphi_{\text{origin}}) \times 1000
\]
\[
Z_{\text{AR}} = -\Delta\text{lat} \times 111{.}320 \times 1000
\]

Het negatieve teken voor de Z-offset is essentieel: in het ARKit-coördinatenstelsel wijst positieve Z naar de camera (achter), terwijl geografisch noorden overeenkomt met negatieve Z. In Swift:

\begin{listing}[H]
    \begin{minted}{swift}
        func worldOffset(from origin: CLLocationCoordinate2D,
        to target: CLLocationCoordinate2D) -> (x: Float, z: Float) {
            let mPerLat = 111_320.0
            let mPerLon = 111_320.0 * cos(origin.latitude * .pi / 180)
            let x = Float((target.longitude - origin.longitude) * mPerLon)
            let z = Float(-(target.latitude  - origin.latitude) * mPerLat)
            return (x, z)
        }
    \end{minted}
    \caption[GPS naar ARKit coördinatenomzetting]{Vlakke-Aarde omzetting van GPS naar ARKit-wereldcoördinaten.}
\end{listing}

De omzettingsfout bedraagt minder dan 0,01\% binnen een straal van 500 meter, ruimschoots voldoende voor de schaal van een begraafplaats. Dit resultaat is in lijn met de bevindingen van \autocite{Xu2018} over de nauwkeurigheid van AR-registratie bij buitentoepassingen op lokale schaal.

Het GPS-oorsprongspunt wordt ingesteld bij de eerste nauwkeurige GPS-fix van de gebruiker. Alle grafposities worden berekend als offsets ten opzichte van dit punt:

\begin{listing}[H]
    \begin{minted}{swift}
        func setOrigin(coordinate: CLLocationCoordinate2D) {
            guard originCoordinate == nil else { return }
            originCoordinate = coordinate
            
            if let grave = targetGrave {

                buildGraveMarker()
            }
        }
    \end{minted}
    \caption[AR oorsprongspunt]{Instellen van het AR-oorsprongspunt bij de eerste GPS-fix.}
\end{listing}

\subsection{De 3D-navigatiepijl}

De navigatiepijl is opgebouwd als een samengesteld \texttt{SCNNode}-object met twee geometrische elementen:

\begin{listing}[H]
    \begin{minted}{swift}
        private func buildArrow() {
            let root = SCNNode()
            
            // Schacht: een platte balk langs de +Z-as
            let body = SCNBox(width: 0.18, height: 0.06,
            length: 0.42, chamferRadius: 0.015)
            body.firstMaterial?.diffuse.contents  = UIColor.systemYellow
            body.firstMaterial?.emission.contents =
            UIColor.systemYellow.withAlphaComponent(0.55)
            let bodyNode = SCNNode(geometry: body)
            
            // Kop: een brede piramide met punt langs +Z.
            // SCNPyramid heeft standaard de punt langs +Y;
            // rotatie van -90 graden rond X heroriënteert naar +Z.
            let head = SCNPyramid(width: 0.38, height: 0.28, length: 0.38)
            head.firstMaterial?.diffuse.contents  = UIColor.systemYellow
            head.firstMaterial?.emission.contents =
            UIColor.systemYellow.withAlphaComponent(0.65)
            let headNode = SCNNode(geometry: head)
            headNode.eulerAngles = SCNVector3(-Float.pi / 2, 0, 0)
            headNode.position    = SCNVector3(0, 0, 0.35)
            
            root.addChildNode(bodyNode)
            root.addChildNode(headNode)
            sceneView.scene.rootNode.addChildNode(root)
            arrowNode = root
        }
    \end{minted}
    \caption[3D navigatiepijl opbouw]{Opbouw van de 3D-navigatiepijl als samengesteld SCNNode-object.}
\end{listing}

De rustoriëntatie is zo gekozen dat de punt wijst langs de \texttt{+Z}-as. De \texttt{SCNPyramid}-geometrie heeft standaard zijn punt langs \texttt{+Y}; door een rotatie van $-90°$ rond de X-as toe te passen wordt deze geheroriënteerd naar \texttt{+Z}. \subsubsection{Per-frame positie- en rotatieupdate}

De pijl volgt de gebruiker: elke renderframe (±60 Hz) wordt de pijl geherpositioneerd op 1,8 meter voor de camera en geroteerd naar het geselecteerde graf. Dit gebeurt rechtstreeks op de SceneKit render thread:

\begin{listing}[H]
    \begin{minted}{swift}
        func renderer(_ renderer: SCNSceneRenderer,
        updateAtTime time: TimeInterval) {
            guard let frame = sceneView.session.currentFrame else { return }
            updateArrow(cameraTransform: frame.camera.transform)
        }
        
        private func updateArrow(cameraTransform: simd_float4x4) {
            guard let arrow = arrowNode else { return }
            
            let cx = cameraTransform.columns.3.x
            let cy = cameraTransform.columns.3.y
            let cz = cameraTransform.columns.3.z
            
            // Kijkrichting: -Z-kolom van de cameramatrix, geprojecteerd op XZ
            let fwdX = -cameraTransform.columns.2.x
            let fwdZ = -cameraTransform.columns.2.z
            let fwdLen = sqrt(fwdX * fwdX + fwdZ * fwdZ)
            let nx: Float = fwdLen > 0.001 ? fwdX / fwdLen : 0
            let nz: Float = fwdLen > 0.001 ? fwdZ / fwdLen : -1
            
            // Positioneer 1,8 m voor de camera, licht onder ooghoogte
            arrow.position = SCNVector3(cx + nx * 1.8, cy - 0.25, cz + nz * 1.8)
            
            // Roteer de +Z-as van de pijl naar het graf
            if let gravePos = graveWorldPosition {
                let dx = gravePos.x - arrow.position.x
                let dz = gravePos.z - arrow.position.z
                arrow.eulerAngles = SCNVector3(0, atan2(dx, dz), 0)
            }
        }
    \end{minted}
    \caption[Per-frame pijlupdate]{Per-frame update van pijlpositie en -rotatie op de SceneKit render thread.}
\end{listing}

Een belangrijk implementatiedetail: de update wordt rechtstreeks op de SceneKit render thread uitgevoerd, zonder \texttt{DispatchQueue.main.async}. Dit vermijdt een vertraging van één of meerdere frames en garandeert vloeiende beweging. De rotatiehoek via \texttt{atan2(dx, dz)} zorgt ervoor dat de \texttt{+Z}-as van de pijl steeds naar het geselecteerde graf wijst, vergelijkbaar met een kompasnaald in 3D.

\subsection{De 3D-grafmarker}

Wanneer de GPS-oorsprong gekend is, wordt een world-locked grafmarker geplaatst op de GPS-positie van het geselecteerde graf:

\begin{listing}[H]
    \begin{minted}{swift}
        private func buildGraveMarker() {
            guard let pos = graveWorldPosition,
            let grave = targetGrave else { return }
            
            let root = SCNNode()
            root.position = SCNVector3(pos.x, 0, pos.z)
            
            // 1. Roterende kubus op grondniveau
            let cube = SCNBox(width: 0.4, height: 0.4,
            length: 0.4, chamferRadius: 0.04)
            cube.firstMaterial?.diffuse.contents = UIColor.systemYellow
            cube.firstMaterial?.transparency     = 0.85
            let cubeNode = SCNNode(geometry: cube)
            cubeNode.position = SCNVector3(0, 0.2, 0)
            
            let spin = CABasicAnimation(keyPath: "eulerAngles.y")
            spin.toValue     = Float.pi * 2
            spin.duration    = 4.0
            spin.repeatCount = .infinity
            cubeNode.addAnimation(spin, forKey: "spin")
            
            // 2. Verticale lichtbalk (6 meter hoog, zichtbaar van veraf)
            let beam = SCNCylinder(radius: 0.03, height: 6.0)
            beam.firstMaterial?.diffuse.contents =
            UIColor.systemYellow.withAlphaComponent(0.3)
            let beamNode = SCNNode(geometry: beam)
            beamNode.position = SCNVector3(0, 3.0, 0)
        \end{minted}
        \caption[3D grafmarker -- deel 1]{Opbouw van de world-locked 3D-grafmarker (deel 1): kubus en lichtbalk.}
    \end{listing}
    
    \begin{listing}[H]
        \begin{minted}{swift}
            // 3. Pulserende grondring
            let ring = SCNTorus(ringRadius: 0.6, pipeRadius: 0.03)
            ring.firstMaterial?.diffuse.contents =
            UIColor.systemYellow.withAlphaComponent(0.5)
            let ringNode = SCNNode(geometry: ring)
            ringNode.position = SCNVector3(0, 0.01, 0)
            
            // 4. Billboard-naambord met naam en levensdatums
            let nameText = SCNText(string: grave.fullName, extrusionDepth: 0.005)
            nameText.font = UIFont.systemFont(ofSize: 0.38, weight: .bold)
            nameText.firstMaterial?.diffuse.contents = UIColor.white
            let nameNode = SCNNode(geometry: nameText)
            
            let bb = SCNBillboardConstraint()
            bb.freeAxes = .Y
            nameNode.constraints = [bb]
            
            root.addChildNode(cubeNode)
            root.addChildNode(beamNode)
            root.addChildNode(ringNode)
            root.addChildNode(nameNode)
            sceneView.scene.rootNode.addChildNode(root)
            graveMarkerNode = root
        }
    \end{minted}
    \caption[3D grafmarker -- deel 2]{Opbouw van de world-locked 3D-grafmarker (deel 2): grondring en naambord.}
\end{listing}

De vier componenten dienen elk een afzonderlijk doel:

\begin{itemize}
    \item \textbf{Roterende kubus} – een semi-transparante gele kubus van 40~cm die langzaam roteert. De rotatie benadrukt het artificiële karakter van het object en maakt het onmiddellijk herkenbaar als een digitaal element.
    \item \textbf{Verticale lichtbalk} – een dunne cilinder van 6 meter hoogte, zichtbaar van tientallen meters afstand. Een verticaal baken dat boven de omgeving uitsteekt biedt een betrouwbaarder visueel referentiepunt dan een ankerpunt op de grond \autocite{Zhao2025}.
    \item \textbf{Pulserende grondring} – een \texttt{SCNTorus} op grondniveau met een pulserende animatie op de buisstraal, die de exacte positie op de grond benadrukt.
    \item \textbf{Billboard-naambord} – een \texttt{SCNText}-object met naam en levensdatums, uitgerust met een \texttt{SCNBillboardConstraint} zodat het altijd naar de camera gedraaid is. De lettertypegrootte van 0,38 wereldeenheden zorgt voor leesbaarheid vanaf circa 15 meter.
\end{itemize}

\subsection{Opacity-beheer bij tracking-instabiliteit}

De \texttt{ARSessionManager} biedt een \texttt{setSceneOpacity(\_:)}-methode die door de \texttt{ViewController} wordt aangeroepen wanneer de AR-trackingkwaliteit wijzigt. Bij tracking-degradatie worden de pijl en de grafmarker gedeeltelijk transparant gemaakt via een vloeiende animatie:

        func setSceneOpacity(_ opacity: CGFloat) {
            SCNTransaction.begin()
            SCNTransaction.animationDuration = 0.4
            arrowNode?.opacity       = opacity
            graveMarkerNode?.opacity = opacity
            SCNTransaction.commit()
        }
    \end{minted}
    \caption[Opacity-beheer AR-nodes]{Aanpassing van de opacity van AR-nodes bij tracking-instabiliteit.}

Het dimmen van de AR-objecten geeft de gebruiker een duidelijk visueel signaal dat de AR-inhoud tijdelijk minder betrouwbaar is, zonder de objecten volledig te verbergen. Wanneer tracking herstelt, wordt de opacity terug op 1.0 gezet.

\section{UI-componenten}

\subsection{ViewController – centrale coördinator}

\texttt{ViewController} fungeert als de centrale coördinator van de applicatie. De view-hiërarchie is als volgt opgebouwd (van laag naar hoog in z-volgorde):

\begin{enumerate}
    \item \textbf{ARSCNView} – de volledige achtergrond, cameraweergave met 3D AR-scène.
    \item \textbf{MinimapViewController.view} – minikaart in de linkerbenedenhoek.
    \item \textbf{GraveInfoCard} – informatiekaart onderaan, zichtbaar bij selectie van een graf.
    \item \textbf{Zoekknop (FAB)} – rechtsbenedenhoek.
    \item \textbf{Trackingstatus-label} – bovenaan gecentreerd, communiceert de AR-trackingkwaliteit.
    \item \textbf{Afstandslabel} – rechtsboven, toont de afstand tot het geselecteerde graf.
    \item \textbf{Fallback-banner} – bovenaan, verschijnt automatisch bij aanhoudende tracking-instabiliteit.
\end{enumerate}

Wanneer de gebruiker een graf selecteert, roept \texttt{ViewController} de centrale methode \texttt{navigateTo(grave:)} aan:

\end{listing}

\subsection{Automatische fallback-banner}
\label{sec:fallback}

De fallback-banner is een oranje melding die automatisch verschijnt wanneer de AR-tracking gedurende meer dan 3 seconden instabiel is \textit{en} de gebruiker actief aan het navigeren is naar een geselecteerd graf. De banner biedt twee opties: de kaart onmiddellijk openen als alternatieve navigatiemodus, of de banner sluiten en de AR-navigatie verder zetten.

\begin{listing}[H]
    \begin{minted}{swift}
        private func startFallbackTimer() {
            guard !bannerManuallDismissed else { return }
            trackingLimitedTimer?.invalidate()
            trackingLimitedTimer = Timer.scheduledTimer(
            withTimeInterval: 3.0, repeats: false) { [weak self] _ in
                self?.showFallbackBanner()
            }
        }
        
        private func showFallbackBanner() {
            guard fallbackBanner.alpha == 0 else { return }
            fallbackBanner.isHidden = false
            UIView.animate(withDuration: 0.35, delay: 0,
            usingSpringWithDamping: 0.8,
            initialSpringVelocity: 0.3) {
                self.fallbackBanner.alpha = 1
                self.fallbackBanner.transform = .identity
            }
            arManager.setSceneOpacity(0.4)
        }
    \end{minted}
    \caption[Fallback banner logica]{Timer-gebaseerde activatie van de fallback-banner bij tracking-instabiliteit.}
\end{listing}

De vertraging van 3 seconden voorkomt dat de banner verschijnt bij korte, tijdelijke tracking-onderbrekingen (zoals het even wegdraaien van de camera). Wanneer de tracking herstelt naar \texttt{.normal}, wordt de banner automatisch verborgen, de timer geannuleerd en de opacity van de AR-nodes teruggezet naar 1.0. De banner verschijnt niet bij de \texttt{.initializing} en \texttt{.excessiveMotion} trackingstatussen, aangezien dit verwachte kortdurende toestanden zijn die geen actie van de gebruiker vereisen.

\subsection{Minikaart (MinimapViewController)}

De minikaart is geïmplementeerd als een child view controller met een \texttt{MKMapView} in satellietmodus. In de standaardtoestand is het een afgerond widget van 120~×~120 pixels in de linkerbenedenhoek. Door op de minikaart te tikken wordt een fullscreen-animatie gestart:

            isFullscreen = true

\section{Iteratief ontwikkelproces}

De implementatie verliep iteratief, overeenkomstig de methodologie. De voornaamste iteraties waren:

\begin{itemize}
    \item \textbf{Iteratie 1 – Basisstructuur:} \texttt{ARSCNView} als achtergrond, minikaart-widget, zoekknop, en runtime-permissies voor camera en locatie.
    \item \textbf{Iteratie 2 – Zoekfunctionaliteit:} live zoeken via \texttt{UISearchController}, \texttt{UISheetPresentationController}, en \texttt{GraveInfoCard}.
    \item \textbf{Iteratie 3 – AR-ankerpunten:} eerste versie van world-locked tekst-annotaties via \texttt{ARAnchor} en \texttt{SCNNode}, geplaatst op GPS-coördinaten.
    \item \textbf{Iteratie 4 – 2D-pijloverlayer:} een platte UIKit-pijl (\texttt{ARDirectionOverlay}) gebaseerd op bearing-berekening. Hoewel functioneel, ontbrak het dieptegevoel van een echte AR-ervaring.
    \item \textbf{Iteratie 5 – 3D AR-pijl:} overgang naar een world-space \texttt{SCNNode} die per frame wordt bijgewerkt. Bevatte correctie van het coördinatenstelsel, de rotatiemathematica en de render thread-implementatie.
    \item \textbf{Iteratie 6 – Visuele verfijning:} hervorming van de pijlgeometrie, vergroting van de grafmarker-labels, en verwijdering van de ARKit feature point-debug-visualisatie.
    \item \textbf{Iteratie 7 – Automatische fallback:} implementatie van de fallback-banner met timer-gebaseerde activatie, opacity-dimming van AR-nodes bij tracking-instabiliteit, en directe koppeling naar de volledigscherm-kaartmodus als alternatieve navigatiemodus.
\end{itemize}

Elke iteratie werd getest op een fysiek iOS-apparaat. ARKit is niet beschikbaar in de iOS-simulator, waardoor fysiek testen onvermijdelijk was voor alle AR-gerelateerde componenten. Problemen die tijdens het testen aan het licht kwamen, werden telkens gedocumenteerd en opgelost in de volgende iteratie. Voorbeelden hiervan zijn de onzichtbare pijl door een verkeerde opbouwvolgorde van \texttt{SCNNode}-kinderen, het latentieprobleem door \texttt{DispatchQueue.main.async} in de render loop, en de onjuiste standaardoriëntatie van \texttt{SCNPyramid} die een expliciete X-rotatie vereiste.

\section{Technische beperkingen en aandachtspunten}

De huidige implementatie heeft een aantal technische beperkingen die relevant zijn voor de evaluatiefase:

\begin{itemize}
    \item \textbf{GPS-nauwkeurigheid:} op een iPhone is de horizontale GPS-nauwkeurigheid doorgaans 3 tot 10 meter. Op een begraafplaats met grafstenen die slechts 60 tot 80 centimeter uit elkaar staan, impliceert dit dat de AR-grafmarker niet altijd exact boven de correcte grafsteen wordt geplaatst. Dit is een inherente beperking van GPS-gebaseerde AR die niet volledig opgelost kan worden zonder aanvullende technologieën zoals UWB-bakens of visuele objectherkenning.
    
    \item \textbf{Initialisatietijd:} het AR-systeem vereist een eerste GPS-fix met een horizontale nauwkeurigheid onder 20 meter voordat de grafmarker geplaatst kan worden. In de praktijk bedraagt deze wachttijd enkele seconden tot maximaal een minuut, afhankelijk van de zichtbaarheid van satellieten op de testlocatie.
    
    \item \textbf{AR world tracking drift:} ARKit's world tracking combineert visuele feature tracking met IMU-data. Bij langdurig gebruik of na onderbrekingen kan de tracking lichtjes driften. De implementatie vangt sessie-onderbrekingen op en herstart automatisch, maar dit impliceert een reset van de tracking en een nieuwe initialisatieperiode.
    
    \item \textbf{Belichting en weersomstandigheden:} heldere zonneschijn of bewolking kunnen de tracking-stabiliteit beïnvloeden. De applicatie communiceert de trackingkwaliteit via een statuslabel en activeert bij aanhoudende instabiliteit automatisch de fallback-banner, die de gebruiker naar de kaartweergave leidt (zie sectie~\ref{sec:fallback}). Een volledig automatische omschakeling zonder gebruikersbevestiging werd bewust niet geïmplementeerd, om de gebruiker de controle te laten over de navigatiemodus.
    
    \item \textbf{Ontbrekende routelijn:} het ontwerp voorzag een virtuele routelijn die de optimale looproute op de kaart aangeeft, vergelijkbaar met het geoloket van de gemeente Beveren. Deze functionaliteit werd niet geïmplementeerd omdat ze een nauwkeurige digitale plattegrond van de looppaden op het kerkhof van Sint-Gillis-Waas vereist. Dergelijke GIS-paddata was niet beschikbaar; de redenen hiervoor worden uitgebreid besproken in sectie~\ref{sec:routelijn}.
\end{itemize}

\section{Niet-geïmplementeerde feature: routelijn op de kaart}
\label{sec:routelijn}

Een van de features die in de ontwerpfase werd voorzien, is een routelijn op de minikaart die de optimale looproute van de gebruiker naar het geselecteerde graf aangeeft. Het geoloket van de gemeente Beveren toont een dergelijke lijn als een oranje gestippelde lijn die de bezoeker langs de bestaande paden leidt. Deze feature werd bewust niet geïmplementeerd in deze proof of concept, om de volgende redenen.

\subsection{Twee soorten routelijnen}

Er bestaat een fundamenteel onderscheid tussen twee soorten routelijnen die visueel op elkaar lijken maar technisch volledig anders zijn:

\begin{enumerate}
    \item \textbf{Rechte lijn (crow flies):} een directe verbindingslijn tussen de gebruiker en het graf, ongeacht obstakels of paden. Dit vereist enkel de twee GPS-coördinaten en is technisch triviaal te implementeren als een \texttt{MKPolyline} op de \texttt{MKMapView}. Deze lijn is echter misleidend: ze suggereert een looproute die dwars door andere grafstenen, bomen en afgesloten zones loopt.
    
    \item \textbf{Geroute lijn (zoals Beveren):} een lijn die de werkelijke looppaden van het kerkhof volgt. Dit vereist een gedetailleerde digitale plattegrond van de beloopbare paden als GIS-data (gewoonlijk in GeoJSON- of Shapefile-formaat), en een routeringsalgoritme dat de kortste weg over dit padennetwerk berekent.
\end{enumerate}

Enkel de tweede variant is zinvol voor een begraafplaatsapplicatie. Een rechte lijn zou de indruk wekken dat de gebruiker de kortste weg volgt, terwijl die in werkelijkheid onbegaanbaar is.

\subsection{Ontbrekende GIS-paddata voor Sint-Gillis-Waas}

De geroute lijn vereist GIS-paddata van het kerkhof van Sint-Gillis-Waas. Uit het interview met de gemeente Sint-Gillis-Waas bleek dat de bestaande digitale data van het kerkhof beheerd wordt via WebBGP, het interne systeem van Cevi. WebBGP bevat graflocaties en concessiedata, maar het is niet zeker of het systeem ook gedetailleerde paddata van het terrein bevat in een formaat dat extern beschikbaar gesteld kan worden.

Zelfs indien dergelijke paddata beschikbaar zou zijn, zou de gemeente deze gezien de juridische en administratieve vereisten die in hoofdstuk~\ref{ch:juridische-analyse} beschreven worden niet zomaar als open data kunnen vrijgeven aan een externe applicatieontwikkelaar. Een formele samenwerkingsovereenkomst en mogelijk een verwerkersovereenkomst zouden vereist zijn.

\subsection{Implicaties voor een productieomgeving}

Voor een productieomgeving die meerdere begraafplaatsen ondersteunt, bijvoorbeeld
verspreid over verschillende Belgische gemeenten of zelfs internationaal, zou het
onpraktisch en onhaalbaar zijn om voor elke begraafplaats afzonderlijk een
samenwerkingsovereenkomst te sluiten met de betrokken gemeente om paddata te
verkrijgen. Dit roept de vraag op of er een schaalbare, gemeente-onafhankelijke
oplossing bestaat.

\textbf{OpenStreetMap als schaalbare databron}

OpenStreetMap (OSM) is een vrij toegankelijke, wereldwijde geografische database
die door vrijwilligers wordt opgebouwd en onderhouden onder een open licentie
(ODbL). OSM ondersteunt specifieke tags voor begraafplaatsen (\texttt{landuse=cemetery})
en de looppaden daarbinnen (\texttt{highway=footway}, \texttt{highway=path}).
Wanneer een begraafplaats voldoende gedetailleerd in kaart is gebracht in OSM,
kunnen de looppaden worden opgevraagd via de Overpass API zonder enige overeenkomst
met een gemeente. Onderzoek bevestigt dat OSM-data voldoende kwaliteit biedt voor
voetgangersroutering op wereldschaal \autocite{Novack2018}.

Dit maakt OSM de meest realistische schaalbare oplossing voor een productieversie
van de applicatie. De routeringslogica, het berekenen van de kortste looproute
over het padennetwerk, kan worden gerealiseerd via bestaande open-source
bibliotheken zoals GraphHopper of OSRM, die beide OSM-data als invoer
ondersteunen.

De praktische beperking blijft echter de \textit{volledigheid} van OSM-data per
begraafplaats. Grote en bekende begraafplaatsen zijn doorgaans goed gedocumenteerd,
maar kleinere gemeentelijke begraafplaatsen zoals die van Sint-Gillis-Waas zijn
in OSM op het moment van schrijven onvoldoende gedetailleerd om betrouwbare
routing te ondersteunen. Dit is geen structureel probleem van de oplossing, maar
een kwestie van databeschikbaarheid die zich in de loop van de tijd verbetert
naarmate de OSM-gemeenschap groeit.

\textbf{Samenvatting van de aanbevolen aanpak voor productie}

\begin{enumerate}
    \item Controleer of de begraafplaats beschikbaar is in OSM met voldoende
    paddetail via de Overpass API.
    \item Indien ja: gebruik OSM-paddata + een open-source routeringsbibliotheek.
    Geen overeenkomst met de gemeente vereist.
    \item Indien nee: werk samen met de gemeente of nodig de lokale OSM-gemeenschap
    uit om de begraafplaats in kaart te brengen. Dit is een eenmalige inspanning
    die de data permanent en vrij beschikbaar maakt voor iedereen.
\end{enumerate}

Deze aanpak bewijst dat de routelijn-feature in een productieomgeving realiseerbaar
is zonder afhankelijk te zijn van individuele gemeentelijke datadelingen, en dat
de ontbrekende GIS-paddata voor Sint-Gillis-Waas een tijdelijke, oplosbare
beperking is en geen fundamenteel obstakel voor het concept.

\section{Platformonafhankelijkheid en overdraagbaarheid}
\label{sec:platform}

De proof of concept werd ontwikkeld als een native iOS-applicatie op basis van ARKit. Deze platformkeuze is gemotiveerd in hoofdstuk~\ref{ch:ontwerp}, maar roept tegelijk een relevante vraag op: is het concept ook realiseerbaar op andere platformen, in het bijzonder op Android-toestellen die wereldwijd een groter marktaandeel hebben? Dit is een essentiële vraag voor de haalbaarheid van een productieomgeving, waarbij de applicatie beschikbaar moet zijn voor een zo breed mogelijk publiek.

Dit onderdeel toont aan dat alle technische componenten van dit proof of concept een directe equivalent hebben op het Android-platform, en dat een cross-platform implementatie technisch haalbaar is zonder de kernfunctionaliteit op te geven.

\subsection{ARCore: het Android-equivalent van ARKit}

Op Android-toestellen wordt AR-tracking verzorgd door \textbf{ARCore}, Google's platform voor het bouwen van augmented reality-toepassingen. ARCore biedt dezelfde drie fundamentele mogelijkheden als ARKit:

\begin{itemize}
    \item \textbf{Motion tracking:} ARCore bepaalt continu de positie en oriëntatie van het toestel in de ruimte via een combinatie van camerabeelden en IMU-sensordata (versnellingsmeter en gyroscoop) \autocite{GoogleARCore2024}.
    \item \textbf{Environmental understanding:} ARCore detecteert horizontale en verticale vlakken in de omgeving, vergelijkbaar met ARKit's plane detection.
    \item \textbf{Geospatial API:} ARCore biedt via de Geospatial API GPS-gebaseerde werelduitlijning, waardoor virtuele objecten op basis van GPS-coördinaten in de echte wereld geplaatst kunnen worden. Dit is exact de functionaliteit die in dit proof of concept gebruikt wordt voor de grafmarker en de navigatiepijl \autocite{Google2023}.
\end{itemize}

De werelduitlijning die in dit proof of concept gebruikt wordt via \texttt{ARWorldTrackingConfiguration.worldAlignment = .gravityAndHeading} (ARKit) heeft een directe Android-tegenhanger: ARCore's \texttt{GeospatialPose} API combineert GPS, kompas en visuele tracking om een uitgelijnde wereldruimte te creëren die overeenkomt met geografische coördinaten. De vlakke-Aarde-conversie van GPS naar lokale AR-coördinaten, beschreven in sectie~\ref{sec:gpsconversie}, is platformonafhankelijke wiskunde die identiek toepasbaar is in een Android-implementatie.

ARCore is beschikbaar op meer dan 1 miljard Android-toestellen die Android 7.0 of hoger draaien \autocite{GoogleARCore2024}. Dit betekent dat het overgrote deel van de actieve Android-smartphones in staat is om dezelfde AR-navigatiefunctionaliteit uit te voeren als het iOS proof of concept.

\subsection{Praktische haalbaarheid}

Onderzoek bevestigt dat ARCore-gebaseerde buitennavigatie op Android dezelfde nauwkeurigheid en gebruikerservaring kan bieden als ARKit op iOS. Zo tonen studies aan dat locatiegebonden AR-navigatiesystemen op Android met ARCore vergelijkbare resultaten behalen als op iOS \autocite{Xu2018}. ARCore is bovendien beschikbaar als SDK voor meerdere ontwikkelomgevingen: native Android (Kotlin/Java), Unity, Unreal Engine en zelfs het web via WebXR \autocite{GoogleARCore2024}.

Voor een productieomgeving zijn er twee realistische implementatiepaden:

\begin{enumerate}
    \item \textbf{Native Android-applicatie} in Kotlin, waarbij alle componenten van dit proof of concept worden herbouwd met ARCore, de Android Location API en de Google Maps SDK. Dit pad biedt de beste prestaties maar vereist een aparte codebase naast de iOS-applicatie.
    \item \textbf{Cross-platform applicatie} via Unity AR Foundation, waarbij één codebase zowel iOS als Android ondersteunt. Dit is de meest efficiënte aanpak voor een productieomgeving met beperkte ontwikkelcapaciteit, ten koste van een lichte performantiereductie ten opzichte van native code.
\end{enumerate}

De conclusie is duidelijk: het proof of concept dat in dit onderzoek werd ontwikkeld, is niet iOS-exclusief van aard. De onderliggende technologie, concepten en navigatielogica zijn volledig overdraagbaar naar het Android-platform, en een cross-platform implementatie is technisch haalbaar met bestaande, volwassen frameworks. De keuze voor iOS in dit proof of concept is een pragmatische beslissing gebaseerd op de beschikbaarheid van ARKit als het meest stabiele AR-framework voor buitenomgevingen, niet een architecturale beperking van het concept zelf.

\section{Aansluiting op methodologie en literatuur}

De implementatie sluit nauw aan bij de doelstellingen uit de methodologie. De twee hoofdcomponenten – kaartweergave en AR-weergave – zijn beide functioneel geïmplementeerd. Vanuit de literatuur zijn de volgende inzichten concreet verwerkt:

\begin{itemize}
    \item \textbf{Verminderde cognitieve belasting \autocite{Zhao2025}:} de integratie van navigatie-informatie in de camerafeed vermijdt dat de gebruiker zijn blik moet wenden naar een apart scherm. De 3D-pijl is altijd zichtbaar in het gezichtsveld.
    
    \item \textbf{AR-registratienauwkeurigheid \autocite{Xu2018}:} de vlakke-Aarde-benadering levert een fout van minder dan 0,01\% binnen 500 meter, wat volstaat voor kerkhofschaal.
    
    \item \textbf{Gebruiksvriendelijkheid voor diverse doelgroepen \autocite{Haekkilae2022}:} de interface is bewust eenvoudig gehouden. Één zoekknop, één kaartwidget en een automatisch verschijnende informatiekaart zijn voldoende voor het basisgebruik, zonder vereiste technische kennis.
    
    \item \textbf{Fallback bij tracking-instabiliteit:} zoals geïdentificeerd als risico in de risicoanalyse (hoofdstuk~\ref{ch:voorbereiding-analyse}), werd een automatisch fallback-mechanisme geïmplementeerd. Bij aanhoudende AR-instabiliteit wordt de gebruiker actief naar de kaartweergave geleid, zodat de navigatie altijd kan worden voortgezet ongeacht de AR-trackingkwaliteit.
\end{itemize}
